# Room Booking Chat Assistant

เอกสารนี้สรุปแนวทางต่อยอดโมดูล **Room Booking** ของ ONE-IL ให้รองรับการสนทนากับ AI Chat ผ่านช่องทางแชต เช่น LINE ในอนาคต โดยยึดระบบที่มีอยู่ใน repo นี้เป็นฐานหลัก

## เป้าหมาย

ให้ผู้ใช้สามารถจองห้องผ่านการสนทนาแบบภาษาธรรมชาติได้ เช่น

- ผู้ใช้ถามห้องว่าง
- AI ตรวจสอบเวลาว่างจากระบบจริง
- AI ถามต่อเรื่องอุปกรณ์หรือรายละเอียดที่ยังขาด
- AI สรุปรายการจองก่อน submit
- ระบบทำการจองจริง
- AI แจ้งผล, ส่งอีเมลยืนยัน, ส่งลิงก์ติดตามสถานะ
- หากพร้อมในเฟสถัดไป ให้เสนอเพิ่มลง Google Calendar

```text
ผู้ใช้: ขอไมค์

AI: วันที่ 21 พ.ค. 2026
AI: เวลา 10.00 - 11.00 น.
AI: อุปกรณ์: Microphone
AI: หากถูกต้อง พิมพ์ "ยืนยันการจอง" ได้เลยครับ
```

## ปุ่มกดใน LINE ทำได้หรือไม่

ทำได้ และควรใช้ด้วยในหลายจุดของ flow นี้

รูปแบบที่เหมาะกับงานจองห้องมี 4 แบบ

- `Quick reply`
ใช้สำหรับคำตอบสั้น ๆ ที่ต้องการให้กดตอบเร็ว เช่น เลือกอุปกรณ์, เลือกช่วงเวลา, ตอบ ใช่/ไม่ใช่

- `Confirm template`
ใช้สำหรับคำถามที่มี 2 ทางเลือกชัดเจน เช่น
  - ยืนยันการจองหรือไม่
  - เพิ่มลง Google Calendar หรือไม่

- `Buttons template`
ใช้เมื่อมีตัวเลือกมากกว่า 2 รายการ แต่ยังไม่ซับซ้อนมาก เช่น เลือกห้องทางเลือก หรือเลือกเมนูถัดไป

- `Flex Message`
ใช้สำหรับสรุปรายการจองให้อ่านง่ายขึ้น เช่น แสดงห้อง, วันเวลา, อุปกรณ์, ปุ่มยืนยัน, ปุ่มดูสถานะ

## มาตรฐานการใช้ปุ่มที่แนะนำ

เพื่อให้ประสบการณ์ใช้งานดีและไม่รก ควรใช้ตามนี้

- ใช้ `quick reply` กับคำถามสั้น ๆ ที่ผู้ใช้ควรตอบทันที
- ใช้ `confirm template` กับจุดที่มีผลต่อการ submit จริง
- ใช้ `Flex Message` สำหรับสรุปรายการสำคัญก่อนและหลังการจอง
- ถ้าต้องเปิดหน้าเว็บ เช่น ดูสถานะการจอง ให้ใช้ปุ่มแบบเปิด URL
- ถ้าระบบใช้เวลาประมวลผล ให้ส่งข้อความสั้นหรือ loading ก่อน แล้วค่อยตอบผล

สรุปแบบใช้งานจริง

- เลือกอุปกรณ์: `quick reply`
- ยืนยันการจอง: `confirm template`
- เพิ่ม Google Calendar: `confirm template`
- ดูสถานะการจอง: ปุ่มเปิด URL
- ให้คะแนนบริการ: ปุ่มเปิด URL หรือ quick reply

## เหตุผลที่ควรทำ

- ลดจำนวนขั้นตอนจากการกรอกฟอร์มหลายช่อง
- เหมาะกับผู้ใช้ที่ต้องการถามเร็วและจองทันที
- ใช้กับมือถือได้ดี โดยเฉพาะผ่าน LINE
- สามารถใช้ AI ช่วยเก็บข้อมูลที่ยังไม่ครบแทนการบังคับกรอกทั้งหมดในครั้งเดียว
- เป็นจุดต่อยอดจากระบบ Room Booking ที่เริ่มเป็นรูปธรรมแล้ว

## สิ่งที่มีอยู่แล้วในระบบปัจจุบัน

จาก repo ปัจจุบัน ระบบมีฐานสำคัญพร้อมต่อยอดแล้ว

- หน้าเช็กห้องและจอง: `/room-booking`
- หน้าดูคำขอของฉัน: `/room-booking/requests`
- หน้าจัดการห้องและอุปกรณ์: `/room-booking/manage`
- RPC สำหรับ submit การจองจริง: `public.submit_room_booking_request(...)`
- ตารางหลัก:
  - `public.reservable_rooms`
  - `public.room_booking_requests`
  - `public.room_booking_equipment_lines`
  - `public.room_default_assets`
  - `public.room_schedule_blocks`
- ระบบแจ้ง approver และส่งอีเมลหลัง submit มีอยู่แล้ว

สรุปคือ AI Chat ไม่ควรสร้าง logic การจองใหม่ทั้งหมด แต่ควรเป็นชั้นสนทนาที่เรียกใช้ backend และ business rules ชุดเดิม

## หลักการออกแบบ

- AI เป็นตัวช่วยสนทนา ไม่ใช่ source of truth
- การเช็กห้องว่างต้องอิงข้อมูลจริงจากฐานข้อมูล
- การจองจริงต้องผ่าน service/backend เดิมของระบบ
- ก่อน submit จริง ต้องมีข้อความสรุปและรอการยืนยันจากผู้ใช้
- ทุกข้อความที่กระทบการจองควร audit ได้
- ถ้าข้อมูลไม่พอ AI ต้องถามเฉพาะช่องที่จำเป็น

## ขอบเขตเฟสแรกที่แนะนำ

เริ่มจากเฉพาะ use case นี้ก่อน

- ผู้ใช้ถามห้องว่างด้วยภาษาธรรมชาติ
- AI แปลงเป็นข้อมูล structured:
  - ห้อง
  - วันที่
  - เวลาเริ่ม
  - เวลาสิ้นสุดหรือระยะเวลา
- ระบบตรวจ availability
- AI เสนออุปกรณ์ที่จองร่วมได้
- AI สรุปคำขอก่อนยืนยัน
- AI ส่งคำขอจองเข้าระบบจริง
- AI แจ้งผลพร้อมลิงก์ติดตามสถานะ
- AI ใช้ปุ่มกดใน LINE เพื่อลดการพิมพ์

ยังไม่ต้องรีบทำในเฟสแรก

- ย้ายวันเวลาแบบหลายรอบในบทสนทนายาวมาก
- การจองหลายห้องในคำสั่งเดียว
- การอนุมัติผ่านแชต
- การแก้ไขคำขอหลัง submit
- การใช้ AI เขียน SQL เอง

## ข้อมูลที่ AI ต้องเก็บให้ครบก่อน submit

ขั้นต่ำควรมี

- ผู้ขอจองคือใคร
- ห้องที่ต้องการจอง
- วันที่จอง
- เวลาเริ่ม
- เวลาสิ้นสุด หรือระยะเวลา
- วัตถุประสงค์การใช้งาน
- จำนวนผู้เข้าร่วม
- อุปกรณ์ที่ต้องการ
- ช่องทางติดต่อกลับ

ค่าที่อาจเติมอัตโนมัติได้จากโปรไฟล์ผู้ใช้

- ชื่อผู้ติดต่อ
- อีเมล
- เบอร์โทร
- employee / user id

## สถานะบทสนทนา (Conversation State)

แนะนำให้แยก state ชัดเจน เพื่อให้ต่อกับ LINE หรือช่องทางอื่นได้ง่าย

1. `collecting_request`
รับข้อมูลการจองจากผู้ใช้

2. `checking_availability`
ตรวจสอบว่าห้องและช่วงเวลาว่างหรือไม่

3. `suggesting_equipment`
แสดงรายการอุปกรณ์ที่ร้องขอได้สำหรับห้องนั้น

4. `awaiting_confirmation`
สรุปรายการจองและรอคำยืนยัน

5. `submitting`
เรียก backend เพื่อ submit คำขอจริง

6. `submitted`
แจ้งผลสำเร็จ พร้อมอีเมลและลิงก์ติดตาม

7. `awaiting_calendar_confirmation`
ถามว่าต้องการเพิ่มลง Google Calendar หรือไม่

8. `handoff_or_error`
กรณีข้อมูลไม่ครบ, ชน policy, หรือระบบขัดข้อง

## Intent ที่ระบบควรรองรับ

- `check_room_availability`
- `select_room`
- `select_time`
- `select_equipment`
- `confirm_booking`
- `decline_booking`
- `cancel_draft_booking`
- `show_booking_status`
- `add_to_google_calendar`
- `decline_google_calendar`

ในทางปฏิบัติ intent เหล่านี้ควร map ไปยัง service function ที่กำหนดตายตัว ไม่ควรปล่อยให้ LLM เดาทางเองทั้งหมด

## Service Function ที่ควรมี

ควรมีชั้น backend สำหรับ AI Chat โดยเฉพาะ เช่น

- `findAvailableRooms(input)`
- `getRoomAvailability(input)`
- `getRequestableEquipmentForRoom(roomId)`
- `buildBookingDraft(input)`
- `submitRoomBookingFromChat(input)`
- `sendBookingConfirmation(input)`
- `buildCalendarEvent(input)`

แนวคิดสำคัญคือ AI เรียก function เหล่านี้ แล้วเอาผลลัพธ์มาสรุปเป็นภาษาคน

## Flow ระบบที่แนะนำ

1. ผู้ใช้ส่งข้อความใน LINE
2. LINE webhook ส่งเข้ามาที่ backend
3. backend ตรวจว่า LINE user นี้ map กับ user ใน ONE-IL คนใด
4. AI/intent layer วิเคราะห์ว่าผู้ใช้ต้องการทำอะไร
5. backend เรียก service ของ Room Booking
6. AI สรุปคำตอบกลับเป็นข้อความไทย
7. ถ้าผู้ใช้ยืนยัน ระบบค่อย submit ผ่าน business logic เดิม
8. ระบบส่งอีเมลยืนยันและตอบลิงก์ดูสถานะกลับไป

## LINE UX Pattern ที่แนะนำ

### 1. ตอนผู้ใช้ถามห้องว่าง

ตอบเป็นข้อความธรรมดาหรือ Flex สั้น ๆ แล้วตามด้วย quick reply

ตัวอย่าง

```text
ห้อง 101 ว่างครับ วันที่ 21 พ.ค. 2026 เวลา 10.00 - 11.00 น.
ต้องการใช้อุปกรณ์อะไรเพิ่มบ้างครับ
```

ตัวเลือกที่แนะนำ

- Microphone
- Notebook
- Projector
- ไม่ใช้อุปกรณ์เพิ่ม

### 2. ตอนรอยืนยันการจอง

ใช้ `confirm template` เป็นหลัก เพราะเป็นจุดที่มีผลต่อการสร้างคำขอจริง

ข้อความตัวอย่าง

```text
ยืนยันการจองห้อง 101 วันที่ 21 พ.ค. 2026 เวลา 10.00 - 11.00 น. ใช่ไหมครับ
```

ปุ่มที่แนะนำ

- `จอง`
- `ไม่จอง`

### 3. หลังจองสำเร็จ

ส่งข้อความสรุปผล แล้วต่อด้วยปุ่มทางเลือกถัดไป

ข้อความตัวอย่าง

```text
ทำการจองให้แล้วครับ
ผมส่งรายละเอียดไปที่อีเมล xxx@xxx.xxx แล้ว
ดูสถานะการจองได้ที่ลิงก์ด้านล่างครับ
```

ปุ่มที่แนะนำ

- `ดูสถานะการจอง`
- `เพิ่มลง Google Calendar`

### 4. ตอนถามเรื่อง Google Calendar

ถ้าต้องการให้ flow ชัดเจน ใช้ `confirm template`

ข้อความตัวอย่าง

```text
ต้องการเพิ่มรายการจองนี้ลง Google Calendar ของคุณหรือไม่ครับ
```

ปุ่มที่แนะนำ

- `ใช่`
- `ไม่`

### 5. ตอนปิดการสนทนา

หากอยากเก็บ feedback หลังจบ flow ให้ถามสั้น ๆ เท่านั้น ไม่ควรถามยาว

ตัวอย่าง

```text
ขอบคุณที่ใช้บริการครับ
หากสะดวก ช่วยให้คะแนนการใช้งานได้ที่ลิงก์นี้ครับ
```

## การเชื่อมกับของเดิมใน ONE-IL

เอกสารนี้ตั้งใจให้ reuse ของเดิมให้มากที่สุด

- ใช้กฎ availability และ overlap เดิมของ Room Booking
- ใช้ RPC `submit_room_booking_request(...)` เดิมเมื่อถึงขั้น submit
- ใช้ข้อมูลห้องจาก `reservable_rooms`
- ใช้ข้อมูลอุปกรณ์จาก `room_default_assets` และ `reservable_asset_settings`
- ใช้หน้าติดตามสถานะเดิม `/room-booking/requests`
- ใช้แนวคิด notification/email ที่มีอยู่แล้วเป็นฐาน

## LINE Integration ที่แนะนำ

ถ้าทำผ่าน LINE ในอนาคต แนะนำองค์ประกอบดังนี้

- LINE Official Account
- Messaging API webhook
- ตาราง map ระหว่าง `line_user_id` กับ `public.users.id`
- ตาราง session สำหรับเก็บ conversation state
- ตาราง chat audit log

สิ่งที่ควรเตรียมเพิ่มสำหรับปุ่มกด

- รองรับ action แบบ `message` สำหรับตัวเลือกทั่วไป
- รองรับ action แบบ `postback` สำหรับการยืนยันที่มีผลต่อระบบ
- รองรับ action แบบ `uri` สำหรับเปิดหน้าสถานะการจอง
- ออกแบบ payload ของ postback ให้สั้น อ่านง่าย และ audit ได้

ตารางที่น่าจะเพิ่ม

- `chat_identities`
- `chat_sessions`
- `chat_messages`
- `chat_action_logs`

## Google Calendar Integration

ความสามารถนี้เหมาะเป็นเฟส 2

- หลังจองสำเร็จ AI ถามผู้ใช้ว่าต้องการเพิ่มลง Google Calendar หรือไม่
- ถ้าผู้ใช้ตอบตกลง ระบบจึงสร้าง event
- ควรเชื่อมแบบ OAuth รายบุคคล ไม่ใช้ calendar กลางแบบ hardcode
- ถ้ายังไม่พร้อมเชื่อม API จริง อาจเริ่มจากส่งไฟล์ `.ics` ทางอีเมลก่อน

รูปแบบ UX ที่แนะนำ

- ถามผู้ใช้หลังจองสำเร็จเท่านั้น
- ถ้าเลือก `ใช่` ค่อยพาไป authorize หรือสร้าง event
- ถ้าเลือก `ไม่` ให้ปิด flow ทันที

ข้อเสนอแนะ

- เฟสแรก: ส่งอีเมลยืนยัน + ลิงก์สถานะ
- เฟสถัดไป: ปุ่ม `เพิ่มลงปฏิทิน`
- เฟสหลังจากนั้น: sync ผ่าน Google Calendar API

## กติกาที่ AI ต้องเคารพ

- ห้ามยืนยันว่า “จองสำเร็จแล้ว” ก่อน backend submit สำเร็จจริง
- ถ้าห้องไม่ว่าง ต้องเสนอทางเลือกที่ใกล้เคียง
- ถ้าห้องต้องใช้อุปกรณ์เฉพาะ ต้องดึงจากข้อมูลจริง
- ถ้าผู้ใช้ยังไม่ยืนยัน ห้ามสร้างคำขอจริง
- ถ้าเจอข้อมูลกำกวม เช่น “พรุ่งนี้บ่าย” ต้อง normalize ก่อน
- ถ้าชน policy เช่น นอกเวลาทำการ, เกิน booking window, หรือเลย minimum advance notice ต้องอธิบายเหตุผลให้ชัด
- ปุ่ม `จอง` ต้องไม่เป็นเพียงข้อความสวยงาม แต่ต้อง map ไป action ที่ backend ตรวจสอบได้
- ถ้าผู้ใช้กดยืนยันหลังเวลาผ่านไปนาน ระบบควร re-check availability ก่อน submit
- ถ้าผู้ใช้กด `เพิ่มลง Google Calendar` ต้องตรวจว่ามีสิทธิ์และมี integration พร้อมใช้งานก่อน

## ตัวอย่างข้อความตอบกลับที่ควรมี

```text
ห้อง 101 ไม่ว่างในช่วง 10.00 - 11.00 น. ครับ

```

```text
ต้องการเพิ่มรายการนี้ลง Google Calendar หรือไม่ครับ
```

## ความเสี่ยงที่ควรวางแผนล่วงหน้า

- ผู้ใช้พูดเวลาคลุมเครือ เช่น “สาย ๆ”, “บ่าย”, “อีก 1 ชั่วโมง”
- LINE user ยังไม่ผูกกับบัญชีในระบบ
- ผู้ใช้ขอห้องด้วยชื่อเล่นที่ไม่ตรงกับ `room_code`
- ขออุปกรณ์ที่ห้องนั้นไม่รองรับ
- backend submit ผ่าน แต่ส่งอีเมลล้มเหลว
- ผู้ใช้ถามต่อหลัง submit เช่น “เปลี่ยนเป็น 2 ชั่วโมง” ซึ่งจริง ๆ อาจต้องสร้าง flow แก้ไขใหม่
- ผู้ใช้กดปุ่มยืนยันซ้ำ
- ผู้ใช้กดปุ่มจากข้อความเก่าที่หมดอายุในเชิงธุรกิจแล้ว
- ข้อมูลใน draft กับ availability ปัจจุบันไม่ตรงกันเมื่อกดปุ่มรอบหลัง

## MVP Definition

ถือว่าเฟสแรกสำเร็จเมื่อทำได้ครบดังนี้

- ผู้ใช้เริ่มจองห้องผ่านแชตได้
- ระบบเช็กห้องว่างจากข้อมูลจริงได้
- ระบบถามข้อมูลที่ขาดได้
- ระบบเสนอรายการอุปกรณ์ได้
- ระบบสรุปรายการพร้อมปุ่มยืนยันได้
- ระบบ submit คำขอจริงเข้าระบบเดิมได้
- ระบบส่งผลลัพธ์พร้อมลิงก์ติดตามสถานะได้
- ระบบถามต่อเรื่อง Google Calendar ด้วยปุ่มได้

## ข้อเสนอการลงมือทำ

ลำดับที่แนะนำ

1. ทำเอกสาร intent + chat state ให้เสร็จ
2. เพิ่ม service layer สำหรับ AI Chat โดยห่อ business logic เดิม
3. ทำ endpoint ภายในสำหรับ chat orchestration
4. ทำตาราง map ผู้ใช้ LINE กับผู้ใช้ระบบ
5. ทำ webhook รับข้อความจาก LINE
6. ทำ quick reply และ confirm template สำหรับ flow หลัก
7. ทำ flow ยืนยันก่อน submit จริง
8. ปิดท้ายด้วย email confirmation, ปุ่มไปหน้าสถานะ, และคำถามเรื่อง Google Calendar

## สรุป

แนวคิดนี้เหมาะมากกับโมดูล Room Booking เพราะระบบหลักมีโครงสร้างพร้อมแล้ว การทำ AI Chat ควรเป็นการเพิ่มชั้นสนทนาและ orchestration ทับบนระบบเดิม ไม่ใช่เขียนระบบจองใหม่อีกชุดหนึ่ง

ถ้าจะเริ่มจริง แนะนำให้เริ่มจาก **LINE + room availability + confirm + submit + status link** ก่อน แล้วค่อยขยายไป Google Calendar และงานหลังบ้านอื่นในเฟสถัดไป
